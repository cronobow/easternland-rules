# 東方山河社區規約修改網站 — 設計規格

**日期**: 2026-03-13
**狀態**: 已確認

---

## 專案背景

東方山河社區需要公告規約修改內容，讓住戶了解：
1. 此次修改所依循的原則
2. 每一條條文修改前後的對比，以及修改備註

網站以單頁靜態方式呈現，方便部署至 GitHub Pages，資料用 Markdown 檔維護。

---

## 技術架構

- **單頁靜態網站**：`index.html` + `app.js` + `style.css`
- **無建置步驟**：直接推上 GitHub，開啟 GitHub Pages 即可
- **外部依賴**：`marked.js`（CDN，解析 Markdown 內文）
- **資料格式**：兩個 `.md` 檔案放在 `data/` 目錄

### 檔案結構

```
house_rule/
├── index.html
├── app.js
├── style.css
├── data/
│   ├── principles.md    # 修改原則
│   └── comparison.md    # 條列比較資料
└── .gitignore
```

## UI 互動功能

| 功能 | 說明 |
|------|------|
| 左側目錄可摺疊 | 預設展開；點按鈕縮窄為 icon 列（僅顯示各條目的條號數字）；再點恢復；手機版（< 768px）側邊欄預設隱藏，改以右下角浮動按鈕開啟 |
| 目錄高亮 | 以 `IntersectionObserver`（threshold 0.5）判定；同時可見多筆時，高亮最靠近 viewport 頂部的一筆；目錄僅列出「條列比較」各筆的標頭 |
| 排序切換 | Toggle 按鈕：「依新規約排序」↔「依舊規約排序」 |
| 條目預設摺疊 | 每筆顯示摘要列，點擊展開三欄完整內容 |
| 點目錄跳轉 | 自動捲動到對應條目並展開 |
| 類型色標籤 | 每筆標頭旁顯示 `修改`／`新增`／`刪除`／`合併` |

## 視覺風格

**簡潔現代**：白底黑字，細線分隔，無裝飾。
三欄比較表色彩區分：
- 修改前：淡灰底
- 修改後：淡藍底
- 備註/原則：淡黃底

類型標籤色彩：
- `修改`：藍色
- `新增`：綠色
- `刪除`：紅色
- `合併`：橙色

## Markdown 資料格式

### `data/principles.md`

```markdown
# 修改原則

## 1. 依照內政部範本調整條文架構
說明內容...

## 2. 精簡重複條文
說明內容...
```

### `data/comparison.md`

每筆記錄以 `---` 分隔，標頭格式：`## [舊:條號][新:條號] 類型`

```markdown
## [舊:2][新:3] 修改

**修改前**
原條文內容...

**修改後**
新條文內容...

**備註/原則**
套用原則2。

---

## [舊:5][新:—] 刪除

**修改前**
已刪除的條文...

---

## [舊:—][新:8] 新增

**修改後**
新增條文內容...

---

## [舊:3,4][新:5] 合併

**修改前**
舊第3條和第4條...

**修改後**
合併後內容...
```

### 解析規則
- JS 以獨立一行的 `---`（前後皆有空行）切分每筆後，需 `.filter(s => s.trim())` 過濾空字串；**條文內容中不得使用 `---` 水平線語法**
- 標頭用正規表達式 `/\[舊:([^\]]+)\]\[新:([^\]]+)\]\s*(.+)/` 解析條號與類型
- `—`（U+2014 EM DASH）代表該側無對應條文（新增或刪除）；JS 解析時對此欄位用 `/^[—–\-]+$/` 多字元比對，容忍人工輸入變體
- 多個舊條號以逗號分隔（合併情況）
- **條號格式限制為正整數**（不支援 `1-1`、`第3條` 等格式）
- 排序：新規約排序依 `新:` 數字；舊規約排序依 `舊:` 第一個數字
- `新:` 欄**僅允許單一條號**（合併類型的新條文只對應一條）
- 合併條目在舊規約排序下以第一個舊條號定位，跳號屬預期行為，不需額外提示
- 無對應條號的項目（`—`）排至該排序方式的最末
- **切換排序後，所有條目重置為摺疊狀態**

### 空欄處理
- 佔位符由 **JS 自動補入**，Markdown 資料檔中不需寫
- `新增` 類型：「修改前」欄 JS 自動補入「（此為新增條文，無修改前內容）」，以斜體灰字呈現
- `刪除` 類型：「修改後」欄 JS 自動補入「（此條文已刪除）」，以斜體灰字呈現
- `合併` 類型：「修改前」欄可能包含多條舊條文，一併呈現
- `備註/原則` 欄為選填；若 MD 中缺少該欄，JS 留空（不補佔位符）

### 錯誤處理
- `fetch()` 使用相對路徑 `./data/principles.md` 和 `./data/comparison.md`（確保在子路徑 repo 下也能正常運作）
- `fetch()` 任一 MD 檔案失敗（404 或網路錯誤）：在對應區塊顯示錯誤提示「資料載入失敗，請重新整理頁面」（紅框提示），不影響另一區塊
- MD 格式錯誤（標頭解析失敗）：跳過該筆，並在 console 輸出警告，不中斷其他項目渲染
- `marked.js` CDN 載入失敗（`typeof marked === 'undefined'`）：在頁面顯示全頁提示「頁面資源載入失敗，請檢查網路後重新整理」

## GitHub Pages 部署

1. 建立 GitHub repo
2. 推上 `main` 分支
3. Settings → Pages → Source: `main` / `root`
4. 完成，網址為 `https://<username>.github.io/<repo>/`

---

## 驗證方式

1. `npx serve .` 或 `python -m http.server` 在本機開啟
2. 確認兩個 MD 檔案正確讀取並 render
3. 切換排序，確認條目順序改變；確認已展開的條目在切換後收起
4. 點擊目錄條目，確認跳轉並展開
5. 摺疊/展開側邊欄正常運作
6. 測試邊界情況：將某一 MD 檔案路徑改錯，確認顯示「資料載入失敗」提示而非白畫面
7. 測試合併條目（`[舊:3,4]`）在兩種排序下的位置是否正確
8. 推上 GitHub 後確認 Pages 可正常存取
